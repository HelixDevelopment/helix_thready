Thready TUI — Terminal Screen Mockups

Bubble Tea + Cobra + Lipgloss, 80-column layouts from wireframes.md §5 (pattern verified against helix_track/llms_verifier/llm-verifier/tui). ANSI roles map to the Thready tokens per lipgloss-theme.md. The TUI defaults to the terminal's dark surface (design-system §7); the light palette serves light terminals via lipgloss.AdaptiveColor. Self-contained — no CDNs, fonts fall back to system mono.

1 · Dashboard wireframes §5

  • Layout is the wireframes §5 dashboard verbatim: left key-hint rail + live pane (WS/SSE) + recent-threads table; the same information as the web Dashboard (§3.3) re-laid for 80 columns.
  • Live rows are processing.progress / completed / failed events; the failed row exposes [r] retry — the idempotent single-claim retry (pressing r twice never double-processes).
  • Domain keys c a k n are layered on the verified key model (§5.1) and chosen not to collide with it; every key is echoed on-screen so the TUI is discoverable without docs.
  • The footer heart is the U+2665 glyph tinted with the Lipgloss Accent style — the locked slogan contract (brand-assets §8.1); attribution is plain text (a terminal renders no logo).

2 · Channels wireframes §3.4 → TUI

  • Rows mirror the web Channels list (§3.4): messenger, auto-detected content type, post count, health. The selected row uses the --surface-warm selection background, not color alone.
  • Max: dev-notes shows the honest ⚠ auth state — the Max adapter is BUILD-NEW [GAP: 5.1 herald]; the row never pretends a session exists.
  • n opens the Add-Channel form (five steps as a prompt sequence, §3.4 wizard → TUI messenger-signin prompt realization, component-library §7.1); esc walks back without losing entered data.
  • Empty state (zero channels) renders the thready-empty contract: one-line reason + the n new primary action — never a blank pane (wireframes §1.1).

3 · Thread view wireframes §5.3 — channel detail → post detail

  • Channel detail lists complete posts (root + organic reply count ↩N) with detected tags and status glyphs ✓ ⭮ ◷ ⚠ — §5.3 verbatim.
  • Post detail splits into the live pipeline pane (ordered steps + the documented precedence, updating on processing.progress) and the replies pane; system replies are hidden and excluded from processing (§3.5/§3.6).
  • Tags distinguish ● direct (accent) from ○ indirect (muted + “AI-derived”) — the distinction is text + glyph, not color-only (a11y contract §6).
  • Generated assets are listed and link out — a terminal cannot show inline video; this is the documented, tested non-parity cell of the component matrix (component-library §7.1).
  • r reprocess / retry-step is the same idempotent single-claim action as every other surface (§3.6).

4 · Search wireframes §5.3 + §3.7 states

  • Query line + mode radio (semantic/keyword/hybrid) + scope checkboxes (≥ 1 enforced on submit) + a scored, keyboard-navigable result list; enter opens Post/Asset detail (§3.7).
  • Scores route through the relational store — vectors are reference-only (§2.1.1); SLO: semantic < 500 ms [OPERATOR §Q14].
  • The second frame is the mandated degraded state: on a HashEmbedder-stub deployment the --warn banner states results are NOT semantic and the score column is hidden [GAP: 2.1] — shown as a separate frame so the default and degraded states are never mixed.
  • Empty state: “No matches — broaden your query” + a clear-filters affordance; a slow embedder yields the graceful “narrow your query” nudge, never a hanging spinner (§3.7).

5 · Processing queue wireframes §3.3 queue + §3.6 precedence · CLI §4.3 events

  • Per-job progress bars over the documented phase order; the precedence caption (download > convert > analyze > research > reply) is the §3.6 contract.
  • The failed row is --danger with an inline [r] retry (idempotent single-claim); errors are never tinted --accent — brand can never mask an error (design-system §6).
  • The events strip is the same stream thready events tail --json consumes: NDJSON, one event per line, reconnect with backoff (CLI §4.3) — TUI and CLI share one Go SDK so behavior is identical.
  • OCR row is flagged [GAP: 2.6]; the queue never claims a stub backend works.
  • Offline: bars freeze at the last known value under a connection drop rather than resetting (the wireframes §6.2 rule, applied to the TUI's WS/SSE subscription).

6 · Login wireframes §5 (Login → Dashboard) + ux-flows §2.1 · textinput ×2 + choice + spinner + statusbar

  • The §5 state machine enters here: Login → Dashboard on "authenticated". Composition: two textinputs (endpoint, token) + a horizontal auth-mode choice + spinner + statusbar.
  • The token path is grounded in CLI §4.1: --endpoint/THREADY_ENDPOINT, --token/THREADY_TOKEN, OS-keychain-first, precedence flag → env → profile file → default — echoed under the field so the TUI never hides where a value came from. TUI and CLI share one Go SDK and one config (~/.config/thready/config.yaml).
  • The device-code frame is the TUI realization of interactive sign-in, mirroring the ux-flows §2.1 pending/failure state machine (spinner while pending; an expired code restarts the flow rather than hanging). Whether the TUI ships device-code or stays scoped-token-only is [OPEN: THREADY-DES-TUI-02].
  • Failure is instructive: the --danger line names the failing value (masked token + expiry date) and the fix line gives the exact command to run — never a bare "invalid". A 401 here is the same condition as CLI exit code 3 (§4.2).
  • The token is masked to its last two characters; secrets are never echoed to the terminal (§6.3 scoped API keys).
  • h/l flips the horizontal choice — the verified "move between areas/tabs" binding (§5.1) reused inside the form row; every key is echoed in the statusbar.

7 · Channel detail wireframes §5.3 + §3.5 → TUI · list + viewport + statusbar

  • This is the thread list within one channel — the §5 state machine's Channels → ChannelDetail → PostDetail middle state. Screen 3 shows the drill-in continuation (Post detail); this screen is the list itself with its scroll viewport.
  • Composition: list (threads) + viewport (the 1–4 of 128 · ▼ 124 more scroll rule) + statusbar.
  • Unread markers: ● in --accent before the title, with the "● unread" legend printed in the scroll rule — glyph + text, never color alone (a11y contract §6). Rows keep the §5.3 anatomy: excerpt, ↩N organic replies, tags, status glyph.
  • The header carries a state chip — ✓ healthy / ⭮ syncing / ⚠ auth needed — the same health vocabulary as the web Channels list (§3.4).
  • The second frame is the honest Max ⚠ auth-warning variant [GAP: 5.1]: a --warn banner, an empty list that says why, and enter opening the thready-messenger-signin prompt (component-library §7.1) — the screen never renders threads a nonexistent session could not have fetched.

8 · Add-Channel form wireframes §3.4 wizard + ux-flows §2 · choice + textinput ×2 + progress + statusbar

  • The §3.4 five-step wizard realized as one TUI prompt sequence (the documented TUI realization, component-library §7.1); the journey and its status-code table are ux-flows §2. Composition: horizontal choice (platform) + textinputs (account, link) + progress + statusbar.
  • The platform picker is a horizontal choice on h/l; Max is visible but honestly marked — the adapter is BUILD-NEW [GAP: 5.1], so it cannot be selected as if it worked.
  • Validation is instructive: the error repeats the failing value (telegram.me/ml_papers) and gives the concrete fix (https://t.me/ml_papers) — a --danger message, never --accent (§1.3); the rule is the §3.4 one (t.me/… or max.ru/join/…, then Resolve must succeed; 422/403 map per the ux-flows §2 table).
  • Resolve success shows the §3.4 resolved preview (title · kind · size) inline; the auto-detected type appears with an override — the user confirms rather than configures (ux-flows §6.2).
  • Submit renders a processing progress line (backfill counted by incoming post.received); the optimistic entry reconciles on the channel.added event (§3.4 states).
  • esc cancels but keeps entered data — the wizard model persists, matching the web wizard's Back/Esc contract (§3.4).

9 · Assets wireframes §3.8 → TUI · list (table) + preview viewport + statusbar

  • Composition: a list rendered as a table — columns type (img/vid/aud/doc), name, size, source post — + a viewport preview pane + statusbar.
  • The selected row uses the inverse-accent Selected style (AccentOn ink on Accent fill, lipgloss-theme §4) — the strongest highlight in the palette, reserved here for the active asset; other lists use the softer --surface-warm selection.
  • Data is consistent with the sibling screens: video-web.mp4/research.md are the p_9f2 generated assets (screen 3); big-buck.mkv is the §5 queue's #movies download; client links resolve through the Asset Service, never raw paths (§7.1).
  • Preview honesty: inline terminal image/video rendering is protocol-dependent (kitty graphics / iTerm2 inline images / sixel) and not universally available, so the pane ships metadata + open/stream and the inline-rendering decision is tracked as [OPEN: THREADY-DES-TUI-01] — the mockup does not pretend a plain terminal shows pixels.
  • r re-download is the web §3.8 broken-physical-link action on the verified r binding; enter opens/streams (Range/HLS via the Asset Service).

10 · Skills wireframes §3.9 → TUI · list + detail viewport + statusbar

  • Composition: list (skills) + viewport detail pane for the selection + statusbar. Columns: enabled ([x]/[ ]), name, version, scope.
  • Vocabulary is identical to the web skills-manager: scopes are the Skill-Graph tiers atomic → composite → umbrella (+ the fallback recipe), and the rows/detail reuse the same records — Research v3 (triggers #Research · GitHub link · IT content; order research > reply; steps web-research(multi-pass) → docs gen → skill-graph grow → semantic idx), Movies v5, Notes/Everything v1.
  • The enabled toggle flips on space (the verified activate binding); enter opens the detail pane — this space/enter split is a Thready layering [DEFAULT — adjustable].
  • Honesty: enabling/disabling edits only the dispatch mapping — the execution engine is BUILD-NEW [GAP: 4.1 helix_skills] (HelixSkills supplies the knowledge DAG, not a runner), and ocr ships disabled because VisionEngine has no OCR engine yet [GAP: 2.6].
  • The fallback recipe is locked on: an uncertain auto-type must always land somewhere for review — never silently dropped [§Q32].
