Features

Everything below exists in the product design and is linked to the screen that shows it. Where a capability depends on a component that is still a scaffold, the page says so — the same honesty bar the rest of this documentation holds itself to.

In depth

Channels & backfill

Add a channel through a five-step wizard with a live resolve preview, then watch the backfill progress post by post. Telegram sessions are part of the shipped design; the Max messenger adapter is in development and is flagged in the UI wherever it applies.

channels.html → [GAP: 5.1 Max adapter]

Threads as they happened

Root posts with their organic reply chains, expandable in place. System status replies are kept separate from human conversation and excluded from reply counts, so a thread reads the way it was written.

thread-explorer.html →

Direct and derived tags

Hashtags you wrote are yours. Tags the AI derives are marked "(derived)" — visually and in the accessible name, never by color alone — so provenance is always one glance away.

post-detail.html →

Skills & recipes

Atomic skills compose into recipes — Research, Movies, Notes — bound to hashtags with an explicit run order and multi-pass steps, testable on a sample post before you commit. The dispatch engine itself is a build-new item, and the design labels it as such.

skills-manager.html → [GAP: 4.1 dispatch engine]

Generated research, with receipts

Research documents carry their sources and pass provenance, link back to the originating post, and export as Markdown or a Docs-Chain PDF. A regenerated document marks the old one stale instead of silently replacing it.

research-viewer.html →

Assets you can trust

Every downloaded asset keeps its sha256, its renditions (including a web-friendly transcode), its linked posts, and a streaming path with Range/HLS. Broken links get a re-download; sensitive files stay locked. OCR for scanned media is tracked as a gap, not sold as done.

assets-browser.html → [GAP: 2.6 OCR]

Live events, in the UI and the terminal

The same event stream — post.received · processing.progress · processing.completed · processing.failed · channel.added — powers the live dashboard, the events monitor, and thready events tail as NDJSON in your terminal. No polling.

events-monitor.html →

Teams, roles, and your own brand

Three-tier RBAC (Root, Account Admin, Standard) with an append-only audit log. Accounts can white-label the product — colors, logo, slogan — behind a server-enforced WCAG-AA gate that rejects inaccessible accents with a measured ratio and a passing suggestion. The Helix Development attribution footer stays, always.

How the pipeline works

1

Ingest

Herald watches your channels and receives every post and reply as it arrives. New channels backfill their history with visible progress.

channel.added → backfill 54/128

2

Process

The Processing Engine classifies each post, then downloads, converts and analyzes its content under a stated precedence, with idempotent retry on any failed step.

download > convert > analyze > research > reply

3

Skills

Hashtags dispatch recipes — composed skills that run research passes, generate documents, and post system replies back into the thread, additively and in order.

#Research → Research v3 (research > reply)

4

Search

Results land in a pgvector semantic index spanning posts, generated docs and assets — semantic, keyword or hybrid, with scores shown only when the full embedder is active.

214 ms · target < 500 ms · degraded mode hides scores

Service names (Herald, Processing Engine, Semantic Search, Event Bus) are the real architecture inventory; the flow above is the same contract rendered in post-detail.html.

How it is run

No third-party CDNs

Fonts and assets are self-hosted; the interfaces load nothing from external CDNs, by design-system contract and CSP posture.

Operated infrastructure

Thready runs on Helix-operated infrastructure: a single dedicated host with three fully separated environments, rootless containers, and secrets that are loaded at runtime only and never logged.

Backed up, measurably

Daily full backups plus hourly database incrementals, with a documented restore runbook — recovery-point objective about one hour, recovery-time objective about four.

open A customer self-host distribution is not currently documented anywhere in the ground truth, so this page does not claim one. Whether "self-host Thready yourself" ever becomes a marketing claim is part of the copy sign-off [OPEN: THREADY-DES-MKT-01] (see README.md).
